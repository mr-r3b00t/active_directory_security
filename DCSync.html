DCSync Attack Hardening Guide

Protect your Active Directory from complete domain compromise by securing directory replication permissions, implementing proper access controls, and establishing robust detection mechanisms.

Understanding The Threat

How DCSync Works

DCSync abuses legitimate Active Directory replication protocols. An attacker with Replicating Directory Changes and Replicating Directory Changes All permissions can impersonate a Domain Controller and request password data for any account, including the KRBTGT account used to sign all Kerberos tickets.

A successful DCSync attack typically means complete and total domain compromise. The attacker can impersonate any user, access any resource, and maintain persistent access indefinitely. Prevention and detection are critical.

1.2 Find All Principals with DCSync Rights

# Import AD module
Import-Module ActiveDirectory

# Get domain distinguished name
$domainDN = (Get-ADDomain).DistinguishedName

# Define the DCSync-related GUIDs
$replicationGUIDs = @{
    'Replicating Directory Changes' = '1131f6aa-9c07-11d1-f79f-00c04fc2dcd2'
    'Replicating Directory Changes All' = '1131f6ad-9c07-11d1-f79f-00c04fc2dcd2'
    'Replicating Directory Changes In Filtered Set' = '89e95b76-444d-4c62-991a-0facbeda640c'
}

# Get ACL on domain object
$acl = Get-ACL "AD:\$domainDN"

# Find principals with replication rights
$dcsyncPrincipals = $acl.Access | Where-Object {
    $_.ObjectType -in $replicationGUIDs.Values -and
    $_.AccessControlType -eq 'Allow'
} | Select-Object @{N='Principal';E={$_.IdentityReference}},
    @{N='Permission';E={
        $guid = $_.ObjectType.ToString()
        ($replicationGUIDs.GetEnumerator() | Where-Object { $_.Value -eq $guid }).Key
    }},
    @{N='Inherited';E={$_.IsInherited}}

$dcsyncPrincipals | Format-Table -AutoSize

1.4 PowerShell Script: Full DCSync Audit

# Comprehensive DCSync permissions audit
function Get-DCSyncPermissions {
    $domainDN = (Get-ADDomain).DistinguishedName
    $domainSID = (Get-ADDomain).DomainSID.Value
    
    # Known safe SIDs (relative to domain)
    $safeSIDs = @(
        "$domainSID-500",  # Administrator
        "$domainSID-502",  # KRBTGT
        "$domainSID-512",  # Domain Admins
        "$domainSID-516",  # Domain Controllers
        "$domainSID-518",  # Schema Admins
        "$domainSID-519",  # Enterprise Admins
        "$domainSID-521",  # Read-Only Domain Controllers
        "S-1-5-9",         # Enterprise Domain Controllers
        "S-1-5-32-544"     # BUILTIN\Administrators
    )
    
    $acl = Get-ACL "AD:\$domainDN"
    $results = @()
    
    foreach ($ace in $acl.Access) {
        $guid = $ace.ObjectType.ToString()
        
        # Check if this is a replication permission
        if ($guid -in @('1131f6aa-9c07-11d1-f79f-00c04fc2dcd2', 
                        '1131f6ad-9c07-11d1-f79f-00c04fc2dcd2')) {
            
            # Try to resolve SID
            try {
                $sid = (New-Object System.Security.Principal.NTAccount($ace.IdentityReference)).Translate(
                    [System.Security.Principal.SecurityIdentifier]).Value
            } catch {
                $sid = "Unknown"
            }
            
            $isSafe = $sid -in $safeSIDs -or $ace.IdentityReference -match 'MSOL_|AAD_'
            
            $results += [PSCustomObject]@{
                Principal = $ace.IdentityReference
                SID = $sid
                Permission = if ($guid -eq '1131f6aa-9c07-11d1-f79f-00c04fc2dcd2') { 
                    'Replicating Directory Changes' 
                } else { 
                    'Replicating Directory Changes All' 
                }
                Inherited = $ace.IsInherited
                Status = if ($isSafe) { '‚úì Expected' } else { '‚ö† INVESTIGATE' }
            }
        }
    }
    
    return $results
}

# Run audit
$auditResults = Get-DCSyncPermissions
$auditResults | Format-Table -AutoSize

# Export to CSV
$auditResults | Export-Csv -Path "DCSync_Audit_$(Get-Date -Format 'yyyyMMdd').csv" -NoTypeInformation

Section 02

Lock Down Replication Rights

Remove unnecessary replication permissions and implement least-privilege access.

2.1 Remove Unauthorized DCSync Permissions

# Remove DCSync permissions from a specific principal
function Remove-DCSyncPermission {
    param(
        [Parameter(Mandatory=$true)]
        [string]$Principal
    )
    
    $domainDN = (Get-ADDomain).DistinguishedName
    $acl = Get-ACL "AD:\$domainDN"
    
    # GUIDs to remove
    $replicationGUIDs = @(
        [GUID]'1131f6aa-9c07-11d1-f79f-00c04fc2dcd2',  # Replicating Directory Changes
        [GUID]'1131f6ad-9c07-11d1-f79f-00c04fc2dcd2'   # Replicating Directory Changes All
    )
    
    $identity = New-Object System.Security.Principal.NTAccount($Principal)
    
    foreach ($guid in $replicationGUIDs) {
        $ace = New-Object System.DirectoryServices.ActiveDirectoryAccessRule(
            $identity,
            [System.DirectoryServices.ActiveDirectoryRights]::ExtendedRight,
            [System.Security.AccessControl.AccessControlType]::Allow,
            $guid
        )
        $acl.RemoveAccessRule($ace) | Out-Null
    }
    
    Set-ACL "AD:\$domainDN" -AclObject $acl
    Write-Host "Removed DCSync permissions from: $Principal" -ForegroundColor Green
}

# Example: Remove from a compromised account
# Remove-DCSyncPermission -Principal "DOMAIN\SuspiciousAccount"

Warning: Test First!

Before removing permissions, verify the account is not legitimately required (e.g., Azure AD Connect, backup solutions). Removing required permissions can break critical services.

2.2 Protect the AdminSDHolder Container

AdminSDHolder protects privileged accounts. Ensure it hasn't been modified:

# Check AdminSDHolder ACL for suspicious entries
$adminSDHolder = "CN=AdminSDHolder,CN=System,$((Get-ADDomain).DistinguishedName)"
$acl = Get-ACL "AD:\$adminSDHolder"

# List all non-inherited ACEs
$acl.Access | Where-Object { -not $_.IsInherited } |
    Select-Object IdentityReference, ActiveDirectoryRights, AccessControlType |
    Format-Table -AutoSize

# Check for any accounts with GenericAll, WriteDacl, or WriteOwner
$dangerousRights = @('GenericAll', 'WriteDacl', 'WriteOwner', 'GenericWrite')
$acl.Access | Where-Object { 
    $rights = $_.ActiveDirectoryRights.ToString()
    ($dangerousRights | Where-Object { $rights -match $_ }).Count -gt 0
} | Select-Object IdentityReference, ActiveDirectoryRights

2.3 Implement Just-In-Time (JIT) Access

Rather than permanent DCSync permissions, use time-limited access:

Option A: Microsoft Identity Manager (MIM) PAM

  • Provides time-bound group membership
  • Requires MIM infrastructure
  • Best for enterprise environments

Option B: Simple JIT Script

# Grant temporary DCSync access (for emergency use only)
function Grant-TemporaryDCSync {
    param(
        [Parameter(Mandatory=$true)]
        [string]$Principal,
        
        [Parameter(Mandatory=$true)]
        [int]$DurationMinutes,
        
        [Parameter(Mandatory=$true)]
        [string]$Reason
    )
    
    # Log the access grant
    $logEntry = [PSCustomObject]@{
        Timestamp = Get-Date
        Principal = $Principal
        Duration = $DurationMinutes
        Reason = $Reason
        GrantedBy = $env:USERNAME
    }
    $logEntry | Export-Csv -Path "C:\Logs\DCSync_JIT.csv" -Append -NoTypeInformation
    
    # Add to a temporary group that has DCSync rights
    Add-ADGroupMember -Identity "Emergency-DCSync-Access" -Members $Principal
    
    Write-Host "Granted DCSync access to $Principal for $DurationMinutes minutes" -ForegroundColor Yellow
    Write-Host "Access will be automatically revoked." -ForegroundColor Yellow
    
    # Schedule removal
    $removeTime = (Get-Date).AddMinutes($DurationMinutes)
    $action = New-ScheduledTaskAction -Execute "PowerShell.exe" -Argument `
        "-Command Remove-ADGroupMember -Identity 'Emergency-DCSync-Access' -Members '$Principal' -Confirm:`$false"
    $trigger = New-ScheduledTaskTrigger -Once -At $removeTime
    Register-ScheduledTask -TaskName "Remove-DCSync-$Principal" -Action $action -Trigger $trigger -Force
}

Protect Privileged Accounts

Attackers typically need to first compromise a privileged account before they can perform DCSync. Protecting these accounts is critical defense-in-depth.

3.1 Protected Users Security Group

Members of this group have additional protections against credential theft:

# Add high-value accounts to Protected Users
$privilegedAccounts = @(
    "Admin-T0-John",
    "Admin-T0-Jane",
    "svc_critical"
)

foreach ($account in $privilegedAccounts) {
    Add-ADGroupMember -Identity "Protected Users" -Members $account
    Write-Host "Added $account to Protected Users" -ForegroundColor Green
}

# Verify membership
Get-ADGroupMember -Identity "Protected Users" | Select-Object Name, SamAccountName

Protected Users can break legacy applications. Test accounts thoroughly before adding to this group. Service accounts may have issues if they rely on NTLM or delegation.

3.2 Enable Credential Guard

Credential Guard uses virtualization-based security to protect credentials:

# Check if Credential Guard is available and enabled
Get-CimInstance -ClassName Win32_DeviceGuard -Namespace root\Microsoft\Windows\DeviceGuard |
    Select-Object AvailableSecurityProperties, SecurityServicesConfigured, SecurityServicesRunning, VirtualizationBasedSecurityStatus

# Enable via Group Policy:
# Computer Configuration ‚Üí Administrative Templates ‚Üí System ‚Üí Device Guard
# ‚Üí Turn On Virtualization Based Security = Enabled
# ‚Üí Credential Guard Configuration = Enabled with UEFI lock

# Or enable via registry (requires Secure Boot and TPM)
reg add "HKLM\SYSTEM\CurrentControlSet\Control\DeviceGuard" /v "EnableVirtualizationBasedSecurity" /t REG_DWORD /d 1 /f
reg add "HKLM\SYSTEM\CurrentControlSet\Control\Lsa" /v "LsaCfgFlags" /t REG_DWORD /d 1 /f

3.3 Disable Credential Caching

Reduce cached credentials on workstations:

# Set cached logons to 0-1 on sensitive systems
# Via GPO: Computer Configuration ‚Üí Windows Settings ‚Üí Security Settings ‚Üí Local Policies ‚Üí Security Options
# "Interactive logon: Number of previous logons to cache" = 0 or 1

# Via Registry
reg add "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon" /v "CachedLogonsCount" /t REG_SZ /d "1" /f

Section 04

Implement Administrative Tiering

Administrative tiering limits the blast radius of compromised accounts by segregating privileges across tiers.

Tier Assets DCSync Risk Controls
Tier 0 Domain Controllers, AD, PKI, Identity Systems Direct risk - these accounts can DCSync PAWs only, no internet, maximum monitoring
Tier 1 Member Servers, Applications, Databases Compromise can lead to Tier 0 attacks Separate admin accounts, jump servers
Tier 2 Workstations, User Devices Common initial compromise point Standard users, limited admin rights

4.1 Tier 0 Isolation

# Create Tier 0 OU structure
$domainDN = (Get-ADDomain).DistinguishedName

# Create Tier 0 OUs
New-ADOrganizationalUnit -Name "Tier 0" -Path $domainDN
New-ADOrganizationalUnit -Name "Accounts" -Path "OU=Tier 0,$domainDN"
New-ADOrganizationalUnit -Name "Groups" -Path "OU=Tier 0,$domainDN"
New-ADOrganizationalUnit -Name "Servers" -Path "OU=Tier 0,$domainDN"
New-ADOrganizationalUnit -Name "PAWs" -Path "OU=Tier 0,$domainDN"

# Create Tier 0 admin group
New-ADGroup -Name "Tier0-Admins" -GroupScope Global -GroupCategory Security `
    -Path "OU=Groups,OU=Tier 0,$domainDN" `
    -Description "Tier 0 administrators with DC and AD access"

4.2 Authentication Policies (Windows Server 2012 R2+)

Restrict where Tier 0 accounts can authenticate:

# Create Authentication Policy for Tier 0
New-ADAuthenticationPolicy -Name "Tier0-AuthPolicy" `
    -Description "Restricts Tier 0 accounts to Tier 0 systems" `
    -UserTGTLifetimeMins 240 `
    -Enforce

# Create Authentication Policy Silo
New-ADAuthenticationPolicySilo -Name "Tier0-Silo" `
    -Description "Tier 0 Authentication Silo" `
    -UserAuthenticationPolicy "Tier0-AuthPolicy" `
    -ComputerAuthenticationPolicy "Tier0-AuthPolicy" `
    -ServiceAuthenticationPolicy "Tier0-AuthPolicy" `
    -Enforce

# Add Tier 0 computers to silo
$tier0Computers = Get-ADComputer -Filter * -SearchBase "OU=Domain Controllers,$domainDN"
foreach ($computer in $tier0Computers) {
    Set-ADComputer -Identity $computer -AuthenticationPolicySilo "Tier0-Silo"
}

# Add Tier 0 users to silo
$tier0Users = Get-ADGroupMember -Identity "Tier0-Admins"
foreach ($user in $tier0Users) {
    Set-ADUser -Identity $user -AuthenticationPolicySilo "Tier0-Silo"
}

5.3 Configure Event ID 4662 for DCSync Detection

Event 4662 requires specific SACL configuration to detect DCSync:

# Add SACL to domain object for replication monitoring
$domainDN = (Get-ADDomain).DistinguishedName
$acl = Get-ACL "AD:\$domainDN"

# Create audit rule for replication rights
$everyone = New-Object System.Security.Principal.SecurityIdentifier("S-1-1-0")

# Audit "Replicating Directory Changes All" access
$auditRule = New-Object System.DirectoryServices.ActiveDirectoryAuditRule(
    $everyone,
    [System.DirectoryServices.ActiveDirectoryRights]::ExtendedRight,
    [System.Security.AccessControl.AuditFlags]::Success,
    [GUID]"1131f6ad-9c07-11d1-f79f-00c04fc2dcd2"  # Replicating Directory Changes All
)

$acl.AddAuditRule($auditRule)
Set-ACL "AD:\$domainDN" -AclObject $acl

Write-Host "SACL configured for DCSync detection" -ForegroundColor Green

5.4 PowerShell Detection Query

# Detect potential DCSync attacks via Event 4662
$startTime = (Get-Date).AddHours(-24)

# DCSync-related GUIDs
$dcsyncGUIDs = @(
    '{1131f6aa-9c07-11d1-f79f-00c04fc2dcd2}',  # DS-Replication-Get-Changes
    '{1131f6ad-9c07-11d1-f79f-00c04fc2dcd2}',  # DS-Replication-Get-Changes-All
    '{89e95b76-444d-4c62-991a-0facbeda640c}'   # DS-Replication-Get-Changes-In-Filtered-Set
)

# Get events from all DCs
$domainControllers = Get-ADDomainController -Filter * | Select-Object -ExpandProperty HostName

foreach ($dc in $domainControllers) {
    Write-Host "Checking $dc..." -ForegroundColor Cyan
    
    try {
        Get-WinEvent -ComputerName $dc -FilterHashtable @{
            LogName = 'Security'
            ID = 4662
            StartTime = $startTime
        } -ErrorAction SilentlyContinue | Where-Object {
            $_.Properties[9].Value -in $dcsyncGUIDs
        } | ForEach-Object {
            $account = $_.Properties[1].Value
            $domain = $_.Properties[2].Value
            $guid = $_.Properties[9].Value
            
            # Filter out legitimate DC replication
            if ($account -notmatch '\$$') {  # Not a computer account
                [PSCustomObject]@{
                    Time = $_.TimeCreated
                    DC = $dc
                    Account = "$domain\$account"
                    Operation = switch ($guid) {
                        '{1131f6aa-9c07-11d1-f79f-00c04fc2dcd2}' { 'Replicating Directory Changes' }
                        '{1131f6ad-9c07-11d1-f79f-00c04fc2dcd2}' { 'Replicating Directory Changes All' }
                        '{89e95b76-444d-4c62-991a-0facbeda640c}' { 'Replicating Directory Changes In Filtered Set' }
                    }
                    Status = 'ALERT - Non-DC Replication Request'
                }
            }
        }
    } catch {
        Write-Warning "Could not query $dc"
    }
}

Section 06

Incident Response

If you detect a DCSync attack, immediate action is required. Assume full domain compromise.

6.1 Immediate Actions

üö®
DCSync Detected - Emergency Response

1. Isolate compromised systems immediately
2. Disable the attacking account
3. Begin password reset planning for ALL accounts
4. Engage incident response team

6.2 KRBTGT Password Reset

Reset KRBTGT twice to invalidate all existing Kerberos tickets:

# KRBTGT reset script - RUN WITH EXTREME CAUTION
# This will invalidate all Kerberos tickets domain-wide

# First reset
Write-Host "Performing first KRBTGT password reset..." -ForegroundColor Yellow
Set-ADAccountPassword -Identity krbtgt -Reset -NewPassword (
    ConvertTo-SecureString (
        [System.Web.Security.Membership]::GeneratePassword(32,8)
    ) -AsPlainText -Force
)
Write-Host "First reset complete. Wait for replication..." -ForegroundColor Green

# Verify replication
repadmin /syncall /APed

# IMPORTANT: Wait at least 10 hours (maximum TGT lifetime) before second reset
Write-Host "WAIT AT LEAST 10 HOURS before the second reset!" -ForegroundColor Red
Write-Host "Monitor for authentication issues during this period." -ForegroundColor Yellow

# Second reset (run after waiting period)
# Set-ADAccountPassword -Identity krbtgt -Reset -NewPassword (...)
‚ö†Ô∏è
KRBTGT Reset Impact

Resetting KRBTGT will cause temporary authentication failures. Plan for: service restarts, user re-authentication, application credential refresh. Test in lab first if possible.

6.3 Full Domain Password Reset

After KRBTGT reset, rotate all privileged account passwords:

# Priority order for password resets:
# 1. All Domain Admins
# 2. All Enterprise Admins  
# 3. All Tier 0 service accounts
# 4. Azure AD Connect account
# 5. All other privileged accounts
# 6. Consider all domain users

# Get accounts that need immediate reset
$criticalAccounts = @()
$criticalAccounts += Get-ADGroupMember -Identity "Domain Admins" -Recursive
$criticalAccounts += Get-ADGroupMember -Identity "Enterprise Admins" -Recursive
$criticalAccounts += Get-ADGroupMember -Identity "Administrators" -Recursive

$criticalAccounts | Select-Object -Unique | ForEach-Object {
    Write-Host "Reset required: $($_.SamAccountName)" -ForegroundColor Red
}
